## 1) Контекст проекта

**Проект:** интерактивная 3D-визуализация математических функций вида `z = f(x, y)` в Blender.  
**Назначение:** учебно-методический STEM-проект (математика + информатика + 3D + Python).

**Текущие ключевые артефакты:**
- `README.md` — общий вход в проект.
- `scripts/visualize_function.py` — быстрая генерация поверхности + материал + `--output` для PNG.
- `scripts/generate_surface_mesh.py` — генератор mesh с пресетами функций.
- `scripts/setup_geometry_nodes_surface.py` — автонастройка Geometry Nodes (амплитуда/частота).
- `docs/` — теория, гайды, эксперименты, шаблоны отчёта/презентации.
- `docs/metodichka/полная_методичка.md` — полный текст методички.
- `docs/metodichka/Методичка.docx` — канонический DOCX методички.
- `docs/STEM_отчет_шаблон_под_скриншоты.docx` — шаблон отчёта с местами под скриншоты.

## 4) Целевое состояние (Definition of Done)

Проект считается доведённым, если одновременно выполнено:

1. **Код**
   - Скрипты стабильно работают в Blender 3.6+ и 4.x.
   - Есть единый слой функций `z = f(x,y)` и параметров.
   - Добавлен пакетный генератор рендеров/кадров.

2. **Артефакты**
   - В репозитории есть `FunctionVisualizer.blend` (подготовленная сцена).
   - Есть минимум 10 качественных рендеров (`assets/renders/`) с читаемыми именами.
   - Есть 1–2 короткие анимации изменения параметров (опционально, но желательно).

3. **Методичка**
   - Полная методичка синхронизирована с реальным состоянием кода/структуры.
   - Эксперименты заполнены реальными наблюдениями и выводами.
   - DOCX-версия чистая, без markdown-артефактов (`**`, лишние символы и т.д.).

4. **Проверяемость**
   - Есть простые тесты/чек-листы, подтверждающие корректность формул и параметров.
   - README позволяет новому пользователю получить результат за 5–10 минут.

## 8) Риски и как обходить

- **Риск:** API Geometry Nodes меняется между версиями Blender.  
  **Решение:** держать совместимость 3.6/4.x через проверку интерфейсов и отдельные ветки логики.

- **Риск:** DOCX-генерация «ломает» формат.  
  **Решение:** один канонический pipeline (либо Pandoc, либо проверенный OOXML-скрипт), затем ручная проверка.

- **Риск:** проект разрастается и теряет учебную простоту.  
  **Решение:** разделять «базовый путь» и «advanced» в документации.

## 12) Вектор развития: формулы + 3D для решения задач (поиск путей и оптимизация)

Следующая итерация должна расширить проект от «визуализации формы» к «решению задач на поверхности».

### 12.1 Что добавить концептуально

1. **Поиск пути на 3D-поверхности**
   - модель поверхности: `z = f(x, y)`;
   - узлы/граф по дискретной сетке;
   - метрика стоимости: длина + уклон + штрафы;
   - алгоритмы: A*, Dijkstra.

2. **Поля стоимости и ограничений**
   - запретные зоны (области недопустимого движения);
   - «дорогие» зоны (высокая стоимость прохода);
   - сценарии: безопасный маршрут, энергоэффективный маршрут, быстрый маршрут.

3. **Визуализация решения в Blender**
   - маршрут как Curve/Polyline поверх поверхности;
   - heatmap стоимости;
   - анимация процесса поиска (опционально).

### 12.2 Учебные кейсы

- Маршрут дрона над рельефом `z = f(x, y)` с ограничением по уклону.
- Поиск безопасного пути «из A в B» с обходом опасных зон.
- Сравнение алгоритмов (A* vs Dijkstra) по длине пути и времени расчёта.

### 12.3 Минимальный технический модуль для этого направления

Рекомендуемая структура:

- `scripts/pathfinding/terrain_graph.py` — дискретизация поверхности в граф.
- `scripts/pathfinding/cost_functions.py` — функции стоимости (длина, уклон, препятствия).
- `scripts/pathfinding/search.py` — A*, Dijkstra.
- `scripts/pathfinding/visualize_path_in_blender.py` — вывод маршрута в Blender.

### 12.4 Критерий готовности этого расширения

Есть демонстрация: пользователь задаёт функцию поверхности + точки старта/финиша, а система строит и отображает маршрут в 3D.
