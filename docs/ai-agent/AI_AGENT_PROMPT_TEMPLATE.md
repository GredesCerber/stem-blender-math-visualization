## 7) СПЕЦИФИКАЦИЯ РАЗВИТИЯ ТЕМЫ ЛАБИРИНТА

Используй это как дизайн-рамку, если пользователь просит углубить практическую часть.

### 7.1 Цель

Показать, что один и тот же 2D-массив может:
- задавать среду (лабиринт);
- быть входом для алгоритмов A*/Dijkstra;
- визуализироваться в Blender;
- использоваться как мост от «игровой» задачи к инженерной навигации.

### 7.2 Единое представление данных

```text
0 = проходимая клетка
1 = стена
```

Требование:
- генерация лабиринта и поиск пути используют **один и тот же массив**, без повторной дискретизации.

### 7.3 Минимальный MVP для модуля лабиринта

1. Генератор лабиринта с `seed` (воспроизводимость).
2. Поиск пути по массиву (A* и/или Dijkstra).
3. Конвертация клеток маршрута в координаты Blender.
4. Визуализация:
   - либо на плоскости;
   - либо как overlay поверх `z = f(x, y)`.
5. Сравнение алгоритмов:
   - длина пути;
   - число посещённых узлов;
   - время.

### 7.4 Критерий готовности

По фиксированному `seed`:

1. лабиринт воспроизводится стабильно;
2. путь воспроизводится стабильно;
3. путь проходит только по `0`-клеткам;
4. визуализация корректно отображает маршрут.

### 7.5 Документирование кейса

При реализации обязательно обновить:

1. `docs\metodichka\полная_методичка.md` (короткий практический блок);
2. `docs\metodichka\методичка_подробная.md` (детальный блок);
3. при необходимости `docs\07_pathfinding_on_surface.md`.
